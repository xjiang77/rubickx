# Rubickx 课程系统总览

## 定位

`rubickx` 不应只被理解为一个教程仓库，也不应只服务于单一主题课程。

更合适的定位是：

> 一个面向 AI coding / agent engineering / agent-native building 的课程系统。

它的目标不是只解释某个工具，而是持续把：
- 代码库
- 工具链
- agent workflow
- 真实项目经验

转化为可学习、可复用、可持续扩展的课程资产。

## 为什么要做成课程系统

如果把 `rubickx` 只当成“某门课”，那么每增加一个主题，目录、导航和产物都会越来越混乱。

如果把 `rubickx` 设计成课程系统，那么：
- 新课程有统一入口
- 新主题可以按相同规范扩展
- docs / web / quiz / visual assets 可以共用结构
- research / dev / knowledge 层更容易接入

## 当前建议的课程主题方向

第一批适合纳入 `rubickx` 的课程包括：
- Claude Code
- Codex
- OpenClaw
- Agent Teams
- Team Protocols
- Worktree Task Isolation
- Harness / Evaluation
- Skills / Tool Use

其中，`Claude Code` 可作为第一门课。

## 课程系统的两层结构

### 1. 平台层
平台层回答：
- 课程放哪里
- 课程元数据如何定义
- 多课程如何导航
- docs-first 与 web-first 如何衔接
- 后续如何持续加课程

### 2. 课程层
每门课至少应包含：
- 课程简介
- 课程大纲
- 模块内容
- 练习 / quiz
- source mapping
- 后续可扩展为 interactive HTML

## docs-first, then interactive

第一阶段建议采用 `docs-first`：
- 先把课程目标、结构、模块与正文做出来
- 再逐步升级为更强交互的 course 页面

这样可以先保证课程内容质量，而不是先陷入页面壳子工程。

## 三类真相分层

### Repo truth
- 实际代码
- walkthrough 源文件
- 测试
- harness

### Course truth
- 课程结构
- 教学叙事
- 学习路径
- quiz / glossary / examples

### Runtime truth
- 构建状态
- 待写模块
- 课程生产流程

这三类内容不应混在一起。

## 当前第一步

当前建议先建立：
1. 课程系统总览
2. 课程设计规范
3. 第一门课：Claude Code 的 brief 和 outline

这会让后续新增课程有统一模板，而不是每次重新设计结构。
