# Rubickx 课程设计规范

## 目标

为 `rubickx` 中的多门课程建立统一的内容组织规范，使课程可以持续增长，而不会在命名、结构和教学方式上失控。

## 每门课程的最低结构

每门课程建议至少包含以下文件：

- `00_课程简介.md`
- `01_课程大纲.md`
- `02_核心机制.md`
- `03_真实工作流.md`
- `04_限制与风险.md`
- `05_练习与问题.md`

若课程较大，可继续拆分更多模块。

## 命名规范

建议目录结构：

```text
go/docs/zh/courses/<course-slug>/
```

例如：

```text
go/docs/zh/courses/claude-code/
```

文件名使用中文序号前缀，便于顺序阅读与后续扩展。

## 课程内容标准

每门课程应尽量满足：
- 面向真实使用场景，而非纯抽象概念
- 从实践进入原理，而不是从术语开始
- 用真实代码 / 真实 workflow 作为讲解材料
- 强调 code ↔ plain language 映射
- 练习题测试应用能力，而不是背诵定义

## 建议的教学风格

- 中文为主体
- 少堆黑话
- 少大段说教
- 多使用：
  - 场景
  - 流程
  - 对比
  - 结构图
  - 代码片段解释

## 从 docs 到 interactive

第一阶段课程以 Markdown docs 为主。

后续若升级为交互式课程页面，应优先保留：
- 原始课程结构
- 明确的模块顺序
- 代码与解释并排的能力
- quiz / visualization 的可插拔空间

## Source mapping 要求

每门课程都应尽量能回答：
- 这部分内容对应哪个 repo / 文件 / 目录
- 哪些代码是 primary source
- 哪些是课程抽象后的解释

这样课程才能长期与代码演进保持对齐。

## 当前执行建议

先用 `Claude Code` 作为第一门课，验证：
- 课程目录结构是否顺手
- 模块拆分是否合理
- 课程表达是否适合未来复制到其他主题
