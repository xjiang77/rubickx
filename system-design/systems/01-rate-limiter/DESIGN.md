## 0. 为什么这题适合开胃

限流器的接口极简（`allow(key) -> bool`），但它逼你直面分布式系统的全部底噪：状态放哪、跨节点怎么一致、计数器原子性、时钟问题、故障时该放行还是拒绝。L5 答到"用 Redis 存计数"就停了；L6 要能讲清**为什么是 Redis、单机内存 vs 集中式的取舍、race condition 怎么治、Redis 挂了怎么办**。

---

## R — Requirements（5min）

### 功能需求

- 限制一个标识（用户 / IP / API key）在时间窗口内的请求数，超出则拒绝。
- 规则可配置：不同 endpoint、不同 tier（免费/付费）有不同阈值。
- 被限流的请求返回 `429 Too Many Requests`，并带 `Retry-After`。

### 非功能需求（L6 主动提，不等面试官问）

| 维度 | 取值 | 为什么重要 |
|------|------|-----------|
| **低延迟** | 限流判断 < 1ms（加在每个请求关键路径上） | 限流器本身不能成为瓶颈 |
| **高可用** | 限流挂了不能拖垮业务 | 决定 fail-open 还是 fail-closed |
| **准确性** | 允许少量误差（边界附近多放/少放几个） | 决定能否用近似算法换性能 |
| **分布式一致** | 多个网关节点共享同一份配额 | 这是本题真正的难点 |
| **可扩展** | 规则数、key 数随业务增长 | 影响存储选型 |

> **L6 关键追问**："你这个限流是**精确**还是**近似**？" —— 大多数场景接受近似（少量超额比拖慢请求好），这个判断决定了后面所有算法选择。

### 边界澄清（主动抛出）

- 限流粒度？per-user / per-IP / per-API-key / 全局，还是组合？
- 限流放在哪一层？客户端 / API Gateway / 每个服务内部？→ **网关层是主流**（集中、对业务透明）。
- 超限是**拒绝**还是**排队**？（拒绝更常见；排队是 leaky bucket 的语义）

## E — Estimation（3min）

假设场景：1 亿用户、10% DAU = 1000 万 DAU，每用户日均 100 次受限请求。

- 日请求 = 1000 万 × 100 = 10 亿/天
- 平均 QPS = 10^9 / 10^5（一天≈10 万秒）≈ **1 万 QPS**
- 峰值 ×5 ≈ **5 万 QPS**

存储：每个 key 的计数器状态 ~100 bytes，活跃 key 1000 万 → ~1 GB。**完全放得进单台 Redis 内存**。这是个关键结论——状态很小，瓶颈在 QPS 和网络往返，不在存储容量。

---

## S — Storage / 状态放哪（5min）

这是限流题的第一个分叉，比"选哪个 DB"更本质：**状态放本地内存还是集中式存储？**

| 方案 | 一致性 | 延迟 | 取舍 |
|------|--------|------|------|
| **本地内存**（每个网关节点各算各的） | 弱（N 个节点 → 实际阈值放大 N 倍） | 最低（无网络往返） | 简单、快，但配额不准 |
| **集中式 Redis** | 强（所有节点共享一份计数） | +1 RTT（~0.5ms 同机房） | 准，但 Redis 成为单点/瓶颈 |
| **本地 + 异步同步**（如 Sliding Window 近似） | 中（最终一致） | 低 | 复杂，适合超大规模 |

- **选型理由（L6 要讲出来）**：选 **Redis** 而不是关系型 DB，因为①状态是高频读写的计数器，关系型 DB 的事务开销和磁盘 IO 扛不住 5 万 QPS；②Redis 单线程模型天然让 `INCR` 原子化，正好解决并发计数；③Redis 有 TTL，窗口过期自动清理，省掉一个清理 job。
- **不选关系型 DB**：每次请求一条 UPDATE + 行锁，几千 QPS 就崩。
- **数据模型**：`key = {user_id}:{endpoint}:{window}`，`value = count`，`TTL = window_size`。

## H — High-level Design（10min）

```
        ┌─────────────┐
Client ─┤ API Gateway │── 命中规则? ──► 调 Rate Limiter 中间件
        └─────────────┘                      │
                                              ▼
                                    ┌──────────────────┐
                                    │ Rate Limiter Mw  │  allow(key)?
                                    │ (算法在这里)      │
                                    └────────┬─────────┘
                                             │ INCR / Lua
                                             ▼
                                    ┌──────────────────┐
                                    │  Redis (计数状态) │
                                    └──────────────────┘
            │ 允许 ──► 转发到后端服务
            │ 拒绝 ──► 429 + Retry-After
```

数据流：请求进网关 → 匹配限流规则（哪个 key、哪个阈值）→ 限流中间件用算法查/更新 Redis 计数 → 允许则放行，拒绝则 429。

规则配置单独存（如配置中心 / Redis 另一个 namespace），网关启动时加载 + 热更新。

## A — API Design（5min）

对外（被限的接口）：标准 HTTP，超限返回：

```
HTTP/1.1 429 Too Many Requests
Retry-After: 12
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1718600000
```

内部限流器接口：

```
allow(key: string, rule: Rule) -> Decision { allowed: bool, remaining: int, retry_after: int }
```

> **L6 点**：返回 `X-RateLimit-*` 头是体贴的 API 设计——让客户端能自我调节、做退避，而不是盲目重试加剧雪崩。

## D — Detailed Design：算法对比（10min，决胜点）

这是 deep dive 的核心。**四种算法**，L6 要能说清每个的适用边界和失败模式。

### 1. Fixed Window Counter（固定窗口计数）

- 思路：把时间切成固定窗口（如每分钟），窗口内 `INCR`，超阈值拒绝，窗口结束清零。
- 优点：实现最简单，内存最小（一个计数器）。
- **致命缺陷 — 边界突刺**：窗口在 00:59 和 01:00 交界，用户可以在 00:59 打满 N 个、01:00 立刻再打满 N 个 → 实际 1 秒内通过 2N。
- 适用：对精度要求低、实现成本敏感的场景。

### 2. Sliding Window Log（滑动窗口日志）

- 思路：存每个请求的时间戳（Redis Sorted Set），每次请求先删掉窗口外的旧时间戳，再数剩下的个数。
- 优点：**完全精确**，没有边界问题。
- 缺点：**内存随请求数线性增长**（每个请求一条记录），高 QPS 下成本爆炸。
- 适用：低频但要求精确的场景（如敏感操作）。

### 3. Sliding Window Counter（滑动窗口计数）⭐ 工程主流

- 思路：固定窗口的近似改良。当前计数 = 当前窗口计数 + 上一窗口计数 × 上一窗口的重叠比例。
  - 例：当前窗口已过 30%，则 `估算值 = 本窗口count + 上窗口count × 70%`
- 优点：平滑了边界突刺，内存仍是 O(1)（两个计数器），近似精度足够好。
- 缺点：近似，假设请求在窗口内均匀分布。
- **这是大厂网关的默认选择**（精度/成本平衡最佳）。

### 4. Token Bucket（令牌桶）⭐ 最常被要求实现

- 思路：桶里以固定速率 `r` 放令牌，容量上限 `b`；每个请求取走 1 个令牌，没令牌就拒绝。
- 关键特性：**允许突发**。桶里攒了 `b` 个令牌时，瞬间能放行 `b` 个请求，之后回落到稳定速率 `r`。
- 两个旋钮：`r`（长期平均速率）、`b`（容忍的突发量）。
- 适用：大多数 API 限流（既限长期速率，又允许合理突发）。这也是 AWS / Stripe 等的主流选择。

### 5. Leaky Bucket（漏桶）

- 思路：请求进队列，以固定速率"漏出"处理。本质是 token bucket 的镜像，但**强制平滑输出**（不允许突发）。
- 适用：需要恒定处理速率的下游（如对接限速第三方、整流流量）。

### 算法选择决策树

```
要精确? ─是─► 请求量大? ─是─► Sliding Window Counter（近似但够准）
  │                └─否─► Sliding Window Log（精确）
  └─否─► 要允许突发? ─是─► Token Bucket
                     └─否─► Leaky Bucket（强制平滑）
默认 API 限流：Token Bucket 或 Sliding Window Counter
```

### Deep Dive：分布式 Race Condition（L6 真正的考点）

多个网关节点并发对同一 key 做"读计数 → 判断 → 写计数"，这是经典的 **check-then-act 竞态**：

```
节点A: GET count=99  ┐
节点B: GET count=99  ┘ 两个都读到 99（阈值 100）
节点A: 判断 99<100，放行，INCR→100
节点B: 判断 99<100，放行，INCR→101  ← 超额放行！
```

**解法（从弱到强）**：

1. **`INCR` 的原子性**：用 `INCR` 返回值判断（先增后判），而非 GET-then-SET。Redis 单线程保证 `INCR` 原子。但 Fixed Window 下"INCR + 设 TTL"两步仍有竞态（第一次 INCR 后还没来得及设 TTL）。
2. **Lua 脚本**：把"读 + 判断 + 写 + 设 TTL"打包成一个 Lua 脚本，Redis 原子执行整个脚本。**这是工业标准做法**——既保证原子性，又省掉多次 RTT。
3. **Redis 单线程 = 天然串行化**：所有节点的请求最终在 Redis 串行执行，根除竞态。代价是 Redis 成为吞吐瓶颈，需分片（按 key 哈希到多个 Redis 实例）。

> **面试金句**："计数器的并发安全不靠应用层加锁，而是下沉到 Redis 的原子操作。GET-then-SET 是 bug，INCR 或 Lua 才是答案。"

## E — Error Handling（3min）

**核心抉择：Redis 挂了，fail-open 还是 fail-closed？**

| 策略 | 行为 | 适用 |
|------|------|------|
| **Fail-open**（放行） | Redis 不可用时所有请求放行 | 多数场景——限流是保护性的，可用性 > 严格限流。宁可短时不限流，也不让限流故障变成全站 503 |
| **Fail-closed**（拒绝） | Redis 不可用时拒绝 | 安全/计费场景——超额比拒绝更糟（如防刷、付费配额） |

补充容错：
- **本地降级缓存**：Redis 抖动时，网关用本地内存做粗粒度限流兜底（牺牲精度保护下游）。
- **Redis 高可用**：主从 + Sentinel / Cluster，避免单点。
- **超时**：调 Redis 设 5-10ms timeout，超时按 fail-open 处理，不能让限流查询本身阻塞请求。

> 用实战经历举例（如果有）：讲一次"依赖组件抖动差点拖垮主链路、靠降级开关救回来"的故事，直接对应 L6 的故障判断力。

## D — Discuss / 扩展（4min，Staff 视角）

- **扩展到全球多机房**：跨机房同步配额成本高（跨区 RTT 几十 ms）。务实做法是**每个机房独立配额**（阈值 = 全局/机房数），接受边界处的近似——一致性换延迟。
- **规则热更新**：限流规则通过配置中心推送，网关本地缓存 + 监听变更，避免每请求查配置。
- **监控**：429 率、Redis 延迟、各 key 命中率——429 突然飙高要能告警（可能是被攻击，也可能是阈值配错）。
- **成本**：Redis 内存 + 网络是主要成本；近似算法（Sliding Window Counter）比精确算法（Log）省一个数量级内存。
- **多级限流**：全局限流 + per-user 限流 + per-endpoint 限流叠加，防止单一维度被绕过。

---

## 三个核心 Trade-off（每个练成 30 秒电梯演讲）

1. **精确 vs 性能/成本**：Sliding Window Log 精确但内存爆炸；Sliding Window Counter 近似但 O(1)。99% 场景选近似——少放几个请求的代价远小于拖慢关键路径。
2. **本地 vs 集中式状态**：本地内存最快但配额放大 N 倍；集中式 Redis 准但加 1 个 RTT 且是瓶颈。超大规模选"本地近似 + 集中式校准"。
3. **Fail-open vs Fail-closed**：限流故障时放行还是拒绝？保护型限流 fail-open（可用性优先），计费/安全型 fail-closed（正确性优先）。这个选择暴露你对"限流到底在保护谁"的理解。
